# Phase 1 — UI Design Contract

> Visual and interaction contract for the Foundation phase (auth pages). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none (raw Tailwind CSS v4.2.2) |
| Icon library | lucide-react 0.577.0 (deferred to Phase 2 -- not needed in Phase 1) |
| Font | Inter (system font stack fallback: `Inter, ui-sans-serif, system-ui, -apple-system, sans-serif`) |

**Rationale:** STACK.md explicitly recommends building from raw Tailwind to teach CSS fundamentals. shadcn deferred to a polish phase. Inter is the default sans-serif for modern web apps and ships with Next.js via `next/font/google`.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding between label and required indicator |
| sm | 8px | Gap between form field label and input, compact padding |
| md | 16px | Default padding inside form inputs, gap between form fields |
| lg | 24px | Card padding, section spacing within auth forms |
| xl | 32px | Gap between form card and page edges on mobile |
| 2xl | 48px | Vertical spacing between logo/heading and form card |
| 3xl | 64px | Page-level vertical centering offset |

Exceptions: none

## Typography

| Role | Size | Weight | Line Height | Usage in Phase 1 |
|------|------|--------|-------------|-------------------|
| Body | 16px | 400 (regular) | 1.5 | Form input text, paragraph text, link text, placeholder text, helper text |
| Label | 14px | 600 (semibold) | 1.4 | Form field labels, error messages |
| Heading | 24px | 600 (semibold) | 1.2 | Page titles ("Log in to Chirp", "Create your account") |
| Display | 32px | 600 (semibold) | 1.1 | App logo/wordmark "Chirp" on auth pages |

**Weights:** Exactly 2 weights are used across all components: `font-normal` (400) for body text, form inputs, helper text, and placeholders; `font-semibold` (600) for headings, labels, button text, and the wordmark. No other weights are permitted in Phase 1.

## Color

| Role | Value | Tailwind Class | Usage |
|------|-------|----------------|-------|
| Dominant (60%) | `#ffffff` (white) | `bg-white` | Page background, form card background |
| Secondary (30%) | `#f9fafb` (gray-50) | `bg-gray-50` | Page background behind card, input field backgrounds |
| Accent (10%) | `#1d9bf0` (Twitter blue) | custom `bg-accent` | Primary CTA buttons ONLY, link text on hover |
| Accent hover | `#1a8cd8` | custom `hover:bg-accent-hover` | Primary CTA button hover state |
| Destructive | `#ef4444` (red-500) | `text-red-500` | Error messages, invalid field borders |
| Border | `#d1d5db` (gray-300) | `border-gray-300` | Form input borders, card borders |
| Border focus | `#1d9bf0` (accent) | `focus:border-accent` | Focused input field borders |
| Text primary | `#0f172a` (slate-900) | `text-slate-900` | Headings, input text, primary body copy |
| Text secondary | `#64748b` (slate-500) | `text-slate-500` | Helper text, placeholder text, secondary labels |

**Accent reserved for:** Primary CTA submit buttons ("Log in", "Sign up"), text links to alternate auth page ("Don't have an account? Sign up"), focused input border ring. Never applied to form field labels, error text, or structural elements.

**Custom CSS variables** (define in `app/globals.css`):
```css
:root {
  --color-accent: #1d9bf0;
  --color-accent-hover: #1a8cd8;
}
```

## Component Inventory (Phase 1)

### Pages

| Page | Route | Layout |
|------|-------|--------|
| Login | `/login` | Centered card on gray-50 background, max-width 400px |
| Signup | `/signup` | Centered card on gray-50 background, max-width 400px |
| Feed (placeholder) | `/feed` | Minimal layout with nav bar containing username and logout button |
| Root | `/` | Redirect only (to `/feed` if authenticated, `/login` if not) |

### Auth Page Layout

Both login and signup share an identical layout shell:

```
[Full viewport height, gray-50 background]
  [Vertically + horizontally centered container]
    [Wordmark "Chirp" -- 32px semibold, slate-900, mb-48px]
    [White card -- rounded-xl, shadow-sm, border gray-300, p-24px, w-full max-w-400px]
      [Heading -- 24px semibold, slate-900, mb-24px]
      [Form fields -- stacked, gap-16px]
      [Primary CTA button -- full width, accent bg, white text, rounded-lg, h-44px]
      [Divider or spacing -- 16px]
      [Link to alternate page -- 14px, accent color, centered]
```

### Form Field Anatomy

Each form field follows this structure:

```
[Label -- 14px semibold, slate-900, mb-8px]
[Input -- 16px regular, h-44px, px-16px, border gray-300, rounded-lg, bg-gray-50]
  [Focus state: border-accent, ring-2 ring-accent/20, bg-white]
  [Error state: border-red-500, ring-2 ring-red-500/20]
[Error message -- 14px semibold, red-500, mt-4px] (only visible when field has error)
```

**Input height is 44px** -- meets touch target accessibility minimum (WCAG 2.5.8).

### Form Validation UX (Claude's discretion decision)

**Approach: On-submit validation with immediate inline feedback.**

- Fields are NOT validated on blur or on change (avoids premature error noise)
- On form submit, all fields are validated simultaneously via Zod schema
- Errors appear as red text below each invalid field (per D-03)
- Errors persist until the form is re-submitted (not cleared on keystroke)
- Server-side uniqueness errors (duplicate email/username) are mapped to the relevant field

**Rationale:** On-submit is simpler to implement, less distracting for users filling forms quickly, and aligns with react-hook-form's default mode.

### Placeholder Feed Page

Minimal page shown after login:

```
[Nav bar -- white bg, border-b gray-300, h-56px, px-16px]
  [Left: "Chirp" wordmark -- 20px semibold, slate-900]
  [Right: "@{username}" label -- 14px regular, slate-500 + "Log out" button -- ghost style, 14px semibold, slate-500 hover:slate-900]

[Main content area -- centered, max-w-600px, pt-48px]
  [Empty state card -- centered text]
    [Heading: "Welcome to Chirp" -- 24px semibold, slate-900]
    [Body: "Your feed is empty. Posting will be available soon." -- 16px regular, slate-500, mt-8px]
```

### Password Strength Requirements (Claude's discretion decision)

**Minimum 8 characters.** No complexity rules (uppercase, special chars, etc.).

**Rationale:** This is a learning project. An 8-character minimum is sufficient to demonstrate password validation without frustrating the developer during testing. The Zod schema enforces this.

### Avatar Placeholder (Claude's discretion decision)

**Not implemented in Phase 1.** No avatar is displayed anywhere in this phase. The `users.avatarUrl` column exists in the schema but is unused. Phase 2 profile pages will introduce avatar display with a colored initial circle (first letter of username) as the default.

## Interaction States

### Button States

| State | Appearance |
|-------|------------|
| Default | `bg-accent text-white font-semibold rounded-lg h-44px` |
| Hover | `bg-accent-hover` (slightly darker) |
| Active/pressed | `bg-accent-hover scale-[0.98]` (subtle press feedback) |
| Disabled/loading | `opacity-60 cursor-not-allowed` + spinner icon replacing button text |
| Focus-visible | `ring-2 ring-accent/30 ring-offset-2` |

### Input States

| State | Appearance |
|-------|------------|
| Default | `border-gray-300 bg-gray-50` |
| Hover | `border-gray-400` |
| Focused | `border-accent ring-2 ring-accent/20 bg-white` |
| Error | `border-red-500 ring-2 ring-red-500/20` |
| Disabled | `opacity-60 cursor-not-allowed bg-gray-100` |

### Loading State

When the login or signup form is submitting:
1. Button text replaced with a 16px CSS spinner (border animation, white color)
2. Button becomes disabled (prevents double-submit)
3. All form inputs become disabled
4. No full-page loading indicator -- submission is fast (local SQLite)

## Responsive Behavior

| Breakpoint | Behavior |
|------------|----------|
| < 480px (mobile) | Card stretches to full width with 16px horizontal margin. No card shadow or border -- card blends with background. |
| >= 480px (tablet+) | Card is centered with max-width 400px, visible shadow and border. |

No other responsive behavior needed -- auth pages are single-column by nature.

## Accessibility Requirements

| Requirement | Implementation |
|-------------|----------------|
| Labels | Every input has a visible `<label>` element with `htmlFor` matching the input `id` |
| Error association | Error messages use `aria-describedby` pointing to error `<span>` with unique id |
| Invalid state | Inputs with errors have `aria-invalid="true"` |
| Focus management | On form submission error, focus moves to the first invalid field |
| Color contrast | All text meets WCAG AA (4.5:1 ratio minimum). Accent blue #1d9bf0 on white = 4.52:1 (AA pass). Red-500 #ef4444 on white = 4.02:1 (used only with structural context, not color alone) |
| Keyboard nav | Tab order follows visual order. All interactive elements focusable. Enter submits forms. |
| Touch targets | All buttons and inputs are minimum 44px tall |
